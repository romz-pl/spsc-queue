## Single Producer, Single Consumer (SPSC) in High Frequency Trading

The **SPSC queue** (typically implemented as a lock-free ring buffer) is one of the most critical data structures in HFT systems. Its guarantee that exactly one thread writes and one thread reads allows it to be implemented **without any locks or atomic CAS operations** — just memory barriers — making it extraordinarily fast.

---

### Where It Appears in an HFT Stack

#### 1. **NIC → Market Data Handler (Kernel Bypass)**
The most latency-sensitive use. After packets arrive via **DPDK or RDMA/OFED** (bypassing the kernel), the NIC's poll-mode driver thread acts as the sole producer, feeding raw packet buffers into an SPSC queue consumed by the market data parsing thread. A lock here would be catastrophic — this path must stay in the **nanosecond range**.

#### 2. **Market Data Parser → Order Book Engine**
The parser decodes raw wire protocol (ITCH, PITCH, SBE) into normalized market events and enqueues them. The order book thread consumes these events to maintain a local **L2/L3 book**. This is a clean SPSC boundary: one parser thread per feed, one book thread per instrument group.

#### 3. **Order Book Engine → Alpha / Signal Engine**
Once the book is updated, a "book changed" event (or a snapshot) is pushed to the signal computation thread. Because signal computation can involve heavier math (regression, ML inference), isolating it behind an SPSC queue lets both threads run at their own pace without stalling each other.

#### 4. **Signal Engine → Order Management System (OMS)**
When the alpha model emits a trading decision, it places an order intent into an SPSC queue consumed by the OMS thread. This decouples decision-making from the mechanics of order construction, risk checks, and FIX/native protocol serialization.

#### 5. **OMS → Exchange Gateway (Outbound)**
The OMS produces fully formed order messages; the gateway thread consumes them and writes to the exchange connection (typically a kernel-bypass TCP stack like **Solarflare OpenOnload** or a co-location FPGA link). Again, a strict SPSC boundary.

#### 6. **Exchange Gateway → OMS (Inbound Execution Reports)**
The reverse path: the gateway receives fills, cancels, and rejects from the exchange and enqueues them for the OMS to process (update positions, P&L, open order state). Separate SPSC queue from the outbound path.

#### 7. **OMS → Risk Manager**
A shadow copy of every order event is pushed to a dedicated risk thread for real-time position/Greeks/exposure computation. This is intentionally a **one-way, best-effort** channel — the risk thread must never slow down the OMS.

#### 8. **Any Thread → Logger**
A high-performance logger (e.g., **NanoLog**, **spdlog async**) uses an SPSC queue per thread. The application thread (producer) writes structured log records into the queue without blocking; a dedicated I/O thread (consumer) flushes to disk. This keeps logging off the critical path entirely.

---

### Why SPSC Specifically (and not MPSC/MPMC)?

| Property | SPSC Benefit in HFT |
|---|---|
| **No CAS / mutex** | Pure loads and stores + a memory fence — sub-10 ns enqueue/dequeue |
| **Cache-line isolation** | Head and tail pointers live on separate cache lines, eliminating false sharing |
| **Wait-free** | Both producer and consumer make progress in bounded time — no starvation |
| **Predictable latency** | No contention means no jitter spikes, critical for tail-latency SLAs |
| **Sequential memory access** | Ring buffer is cache-friendly; hardware prefetcher works perfectly |

---

### The Canonical Implementation

The structure is almost always a **power-of-two ring buffer** where:
- The **producer** writes to `buffer[head & mask]` then increments `head` with a **release store**.
- The **consumer** reads from `buffer[tail & mask]` then increments `tail` with a **release store**.
- No locks, no atomics beyond the fence — just `std::atomic` with `memory_order_acquire/release` (or even `memory_order_relaxed` with an explicit `std::atomic_thread_fence` on x86, where the TSO memory model makes stores implicitly ordered).

In essence, the entire HFT pipeline is often architected as a **chain of SPSC queues**, one per thread boundary, forming a deterministic, lock-free assembly line from raw wire bytes to an order on an exchange.
