# HFT Pipeline, Stage 1

Stage 1 is the most critical boundary in the entire HFT system — the moment raw electrical signals crossing a wire become structured financial data that a strategy can act on. Every architectural decision here is driven by a single constraint: eliminating latency at any cost.

---

## The latency problem with conventional networking

In a standard Linux environment, an incoming network packet follows a long bureaucratic path before any application code sees it. The NIC raises a hardware interrupt, the kernel's interrupt handler wakes up, the packet is copied through a socket buffer, and finally a system call hands it to user-space. This entire journey — context switches, memory copies, scheduler overhead — consumes anywhere from **2 to 10 microseconds**. In HFT, where a price edge might exist for only a few hundred microseconds, this overhead alone can make the difference between a profitable trade and a missed opportunity.

---

## Kernel bypass: the architectural solution

The solution is to eliminate the kernel entirely from the data path. Frameworks like **DPDK** (Data Plane Development Kit) and **RoCE** (RDMA over Converged Ethernet) achieve this by mapping the NIC's packet buffers directly into user-space memory via DMA. The application holds a pointer directly into the hardware's receive descriptor ring — no copies, no interrupts, no system calls.

## The PMD thread: a dedicated CPU core as a wire-tap

The poll-mode driver thread is the soul of this stage. Rather than waiting for the NIC to raise an interrupt (which would incur context-switch overhead), the PMD thread runs an **infinite busy-poll loop**, calling `rte_eth_rx_burst` in DPDK to drain the NIC's receive descriptor ring as fast as possible. The thread is pinned to an isolated CPU core — configured via `isolcpus` in the kernel boot parameters — meaning the Linux scheduler will never preempt it or run other tasks on that core. The thread achieves something close to a **hardware wire-tap**: it is always watching, with no delay between a packet's arrival and the start of processing.

This comes at a real cost: one full CPU core is permanently 100% utilized, even during quiet markets. This is a deliberate engineering trade-off — the alternative (interrupt-driven handling) would introduce latency spikes of 2–10 µs every time the kernel gets involved, which is unacceptable.

---

## The SPSC queue: more than just a data structure

The lock-free Single-Producer Single-Consumer queue between the PMD thread and the parser thread is deceptively important. Because there is exactly one writer and one reader, the queue requires **no mutexes or atomic compare-and-swap operations** on the fast path — just cache-line-aligned head/tail indices, which can be updated with simple sequential stores and loads. This is structurally impossible to deadlock.

Beyond thread communication, the SPSC queue plays two additional roles that the description rightly highlights. First, it acts as a **back-pressure valve**: if the market data parser temporarily falls behind (say, a burst of thousands of quotes arrives simultaneously during a market open), packets queue up safely rather than being dropped, and the parser drains the backlog when the burst subsides. Second, during quiet periods, the PMD thread can **batch multiple packets** into the queue before the parser wakes on the next poll cycle, amortizing per-item overhead.

The queue itself must be carefully engineered: false sharing between the producer's head pointer and the consumer's tail pointer would cause them to bounce cache lines between CPU cores, adding tens of nanoseconds per operation. Proper implementation pads each pointer to a full 64-byte cache line.

## Software development implications

Building this stage correctly demands expertise well outside typical application development:

**Systems programming depth** — This code is written in C or C++ (occasionally Rust for newer systems). There are no safe abstractions here. The PMD thread directly dereferences hardware memory-mapped registers, manages memory-mapped packet buffer pools (mbufs in DPDK), and must handle descriptor ring wraparound without error.

**OS configuration as code** — The software is only half the picture. The deployment environment requires disabling CPU frequency scaling (to eliminate `p-state` latency jitter), configuring huge pages (2 MB or 1 GB) to reduce TLB pressure on the hot path, disabling the NIC's hardware interrupt coalescing, and pinning IRQ affinity away from the PMD core.

**Measurement discipline** — Latency at this layer is measured in nanoseconds using hardware timestamps from the NIC itself (not `gettimeofday()` or `clock_gettime()`, which have their own overhead). Engineers use tools like `rdtsc` (the CPU's timestamp counter) and specialized NIC timestamping features (e.g., Mellanox/NVIDIA ConnectX hardware timestamps) to characterize the exact latency distribution, including tail latencies at the 99.9th percentile.

## Business implications

Stage 1 defines the firm's **information latency floor** — the minimum time between a market event existing in the world and the firm knowing about it. No amount of clever strategy or fast execution can compensate for being slow here. A competitor who receives the same exchange feed 500 ns earlier can react before your system has even decoded the message.

This also drives significant **capital expenditure**. Kernel-bypass-capable NICs (Mellanox/NVIDIA ConnectX, Solarflare/Xilinx) cost substantially more than commodity hardware. Co-location in exchange data centers (to minimize the physical cable distance and thus propagation delay) can cost tens of thousands of dollars per month per rack. The entire infrastructure is sized around the assumption that Stage 1 latency must be sub-200 ns end-to-end, making it one of the most expensive per-line-of-code systems in existence.

Finally, the design imposes a hard **operational constraint**: this stage must never block, sleep, or allocate memory in the hot path. A single call to `malloc` on a cache miss could stall the PMD thread for hundreds of nanoseconds. All buffer pools are pre-allocated at startup, which means the system's memory footprint is fixed and predictable, but capacity planning must be done conservatively upfront.
