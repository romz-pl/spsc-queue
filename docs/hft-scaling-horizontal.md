## Horizontal Scaling for HFT with SPSC

Horizontal scaling in HFT means distributing the trading pipeline across **multiple physical machines** while preserving the ultra-low latency and determinism that SPSC queues provide within a single node. The core challenge is that SPSC is inherently an **intra-process, shared-memory construct** — crossing a machine boundary means replacing it with a network equivalent that mimics its properties as closely as possible.

---

### 1. The Fundamental Problem: SPSC Doesn't Cross Wires

Within one machine, an SPSC queue is a ring buffer in shared memory. The producer writes, the consumer reads — no serialization, no syscalls, no network. The moment you add a second machine, you must:

- **Serialize** the message (or use zero-copy)
- **Transmit** it over a physical medium
- **Deserialize** on the other side
- **Enqueue** into a local SPSC queue for the consuming thread

The goal of horizontal scaling is to make this inter-node path **as close to a local SPSC enqueue as physically possible**, and to structure the workload so each machine has clean, well-defined SPSC pipelines internally.

---

### 2. Network Fabric: The Inter-Node SPSC Equivalent

#### RDMA — The Closest Network Analog to SPSC

Remote Direct Memory Access (RDMA) is the key technology. With **one-sided RDMA writes**, a producer on Machine A can write directly into a pre-registered memory region on Machine B — bypassing both machines' kernels and CPUs on the receive side.

```
Machine A                              Machine B
─────────────────                      ─────────────────
[Producer Thread]                      [Ring Buffer in RAM]
      │                                       ▲
      │  RDMA Write (one-sided)               │
      └──────────── HCA ──── IB Fabric ───── HCA
                    (no CPU involvement on B's receive path)
```

This is the closest network-level analog to a local SPSC enqueue:
- **No syscall** on either side (after setup)
- **No copy** — data lands directly in the target buffer
- **No consumer CPU involvement** on receive — it just polls the tail pointer
- Latency: **~1–2 µs** over InfiniBand HDR (200 Gbps)

The receiver still uses a local SPSC queue — RDMA just becomes the mechanism by which the remote producer populates it.

#### Technology Comparison

| Transport | Latency | CPU Involvement | Kernel Bypass | SPSC Analog Quality |
|---|---|---|---|---|
| **RDMA one-sided write** | ~1–2 µs | Near zero on RX | Yes | ★★★★★ |
| **RDMA two-sided send/recv** | ~2–3 µs | Low | Yes | ★★★★☆ |
| **DPDK UDP (userspace)** | ~3–5 µs | Moderate | Yes | ★★★☆☆ |
| **Solarflare OpenOnload TCP** | ~5–10 µs | Low | Yes | ★★★☆☆ |
| **Standard kernel TCP** | ~50–200 µs | High | No | ★☆☆☆☆ |
| **Shared memory (same host)** | ~50–100 ns | None | N/A | ★★★★★ |

### 3. Horizontal Decomposition Patterns

#### Pattern 1: Feed Handler Farm → Aggregation Tier

The most common horizontal split. Market data from dozens of exchanges cannot be handled by one machine within latency budgets. You deploy a **farm of feed handler machines**, each dedicated to one or a few exchanges:

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Feed Handler 1 │     │  Feed Handler 2 │     │  Feed Handler 3 │
│  (NYSE/ARCA)    │     │  (NASDAQ)       │     │  (CME Futures)  │
│                 │     │                 │     │                 │
│ NIC→SPSC→Parser │     │ NIC→SPSC→Parser │     │ NIC→SPSC→Parser │
│ →SPSC→Book      │     │ →SPSC→Book      │     │ →SPSC→Book      │
└────────┬────────┘     └────────┬────────┘     └────────┬────────┘
         │   RDMA multicast      │                        │
         └──────────────┬────────┘────────────────────────┘
                        ▼
              ┌─────────────────┐
              │  Alpha / Signal │
              │  Machine        │
              │                 │
              │ RDMA→SPSC→Model │
              │ →SPSC→OMS       │
              └─────────────────┘
```

Each feed handler machine has clean internal SPSC pipelines. The inter-machine path uses RDMA or multicast UDP, feeding into a local SPSC queue on the receiving end.

#### Pattern 2: Instrument Sharding

For markets with thousands of instruments (equities, options), horizontally shard by **instrument group**. Each machine owns a subset of symbols end-to-end:

```
Shard Machine 1: AAPL, MSFT, GOOG  → Full pipeline (book→signal→OMS)
Shard Machine 2: AMZN, TSLA, NVDA  → Full pipeline (book→signal→OMS)
Shard Machine 3: SPX options chain  → Full pipeline (book→signal→OMS)
```

Within each shard machine, the pipeline is a chain of SPSC queues. Cross-shard communication (e.g., for pair trades or basket strategies) goes over RDMA with dedicated inter-shard SPSC queues at each node.

#### Pattern 3: Strategy Separation

Different strategy types have different latency and compute profiles. Separate them onto dedicated machines:

```
┌──────────────────────┐      ┌──────────────────────┐
│  Market Making Box   │      │  Statistical Arb Box │
│  (ultra-low latency) │      │  (more compute)      │
│  SPSC chain < 500ns  │      │  SPSC chain < 5µs    │
└──────────────────────┘      └──────────────────────┘
         │                               │
         └──────────┬────────────────────┘
                    ▼
         ┌──────────────────┐
         │  OMS / Risk Box  │
         │  (consolidation) │
         └──────────────────┘
```

This ensures a heavy signal computation on the stat-arb box never causes jitter on the market-making box's SPSC pipeline.

#### Pattern 4: OMS / Risk as a Dedicated Tier

The OMS and risk engine are often separated onto their own machines, connected via RDMA to all strategy machines. Each strategy machine has an outbound SPSC queue whose consumer is a RDMA writer; the OMS machine has an inbound SPSC queue populated by RDMA from all strategy machines (now MPSC at the network level, but locally an SPSC per source):

```
Strategy Box 1 ──RDMA──► ┌─────────────────────────┐
Strategy Box 2 ──RDMA──► │  OMS / Risk Machine     │
Strategy Box 3 ──RDMA──► │                         │
                         │ [Per-source SPSC queues]│
                         │ → Risk Engine           │
                         │ → Exchange Gateways     │
                         └─────────────────────────┘
```

The OMS machine converts the multi-source RDMA ingress into **one SPSC queue per source**, maintaining the single-consumer guarantee locally even though the network view is many-to-one.

### 4. RDMA Ring Buffer: Replicating SPSC Over the Network

The most sophisticated pattern is implementing a **distributed SPSC ring buffer** using one-sided RDMA. This directly maps the local SPSC model onto a network link:

```cpp
// On the producer machine:
struct RemoteSPSCProducer {
    ibv_qp*    qp;           // RDMA queue pair
    uint64_t   remote_addr;  // Remote ring buffer VA
    uint32_t   remote_rkey;  // Remote memory key
    uint64_t   head;         // Local head counter
    uint64_t   mask;         // buffer_size - 1
    
    void enqueue(const Message& msg) {
        uint64_t slot = head & mask;
        
        // One-sided RDMA write — no CPU on remote side
        rdma_write(
            qp,
            &msg, sizeof(msg),           // local source
            remote_addr + slot * sizeof(Message),  // remote dest
            remote_rkey
        );
        
        // Write the new head value to remote tail-check location
        rdma_write_imm(qp, ++head, remote_head_addr, remote_rkey);
    }
};

// On the consumer machine — identical to a local SPSC consumer:
void consumer_loop() {
    while (true) {
        if (local_head.load(acquire) != tail) {
            process(buffer[tail & mask]);
            tail.fetch_add(1, release);
        }
        // Spin — no syscall, no lock
    }
}
```

The consumer on the remote machine has no idea whether its SPSC producer is local or remote — it just polls the head pointer as usual. The RDMA fabric is entirely transparent to it.

### 5. Multicast for Market Data Fan-Out

When the same market data must reach multiple strategy machines, **UDP multicast with a kernel-bypass stack** (DPDK or OpenOnload) is used rather than N independent RDMA unicast streams:

```
Feed Handler Machine
        │
        ├── Multicast Group 239.x.x.1  (equity feeds)
        └── Multicast Group 239.x.x.2  (futures feeds)
                │                  │
        Strategy Box 1      Strategy Box 2
        NIC → SPSC →        NIC → SPSC →
        Parser Thread       Parser Thread
```

Each receiving machine's NIC interrupt (or poll-mode driver) is the SPSC producer for that machine's internal pipeline. Multicast allows **one transmission, many receivers** without the feed handler being a bottleneck.

For ultra-low latency, **hardware multicast on InfiniBand** (IB multicast) is preferred — it replicates at the switch level, adding essentially zero latency versus unicast.

### 6. Co-location & Physical Topology

Horizontal scaling in HFT is inseparable from **physical proximity**. The speed of light imposes a hard floor: every 300km of fiber adds ~1ms one-way latency.

#### Rack-Level Topology
Within a co-location facility, machines in the same rack communicate over **direct-attach copper (DAC)** cables — sub-100ns at the physical layer. Your horizontal cluster should be:

```
Same rack:      Feed Handlers + Alpha Machine + OMS   (< 200ns RTT)
Adjacent rack:  Risk / Compliance Machine             (< 500ns RTT)
Same cage:      Backup / DR machines                  (< 1µs RTT)
```

#### Cross-Exchange Arbitrage: Microwave & Laser Networks
When horizontal scaling spans **geographic locations** (e.g., NYSE in Mahwah, NJ and CME in Aurora, IL), the inter-machine SPSC-equivalent link is a **microwave or free-space laser network**:

| Link | Distance | Latency | vs Fiber |
|---|---|---|---|
| **Fiber** | ~1200 km | ~4.5 ms | Baseline |
| **Microwave** | ~1200 km | ~4.09 ms | ~8% faster |
| **Laser (FSO)** | ~1200 km | ~4.0 ms | ~10% faster |

The machines at each end of these links still use internal SPSC pipelines. The microwave link becomes the "wire" between the consumer of one node's SPSC output queue and the producer of the remote node's SPSC input queue.

### 7. Clock Synchronization Across Nodes

SPSC within a machine can use `RDTSC` and ignore wall time. Across machines, you need **synchronized clocks** or your sequencing and latency measurements are meaningless.

#### PTP / IEEE 1588
Hardware-timestamping NICs synchronized via **Precision Time Protocol** achieve **sub-100ns clock accuracy** across machines in the same co-location:

```bash
# Run PTP on the primary NIC (hardware timestamping)
ptp4l -i eth0 -H -m   # hardware mode
phc2sys -s eth0 -c CLOCK_REALTIME -O 0 -m
```

#### GPS-Disciplined Oscillators
For the most demanding setups, each machine has a **GPS receiver and atomic oscillator** (GPSDO) feeding a PPS (pulse-per-second) signal directly to the NIC's hardware clock. This gives **< 10ns accuracy** independent of network conditions — critical when sequencing events across geographically separated SPSC pipeline endpoints.

### 8. Failure Isolation and Redundancy

Horizontal scaling also enables **fault domains** — something impossible with a single vertical machine:

```
Primary Cluster          Shadow Cluster
────────────────         ──────────────────
Feed Handler A  ─RDMA──► Feed Handler A'
Alpha Box       ─RDMA──► Alpha Box'
OMS / Gateway            OMS / Gateway'
```

Each machine publishes its SPSC output over RDMA to both the primary next-stage machine and a shadow. The shadow maintains a **warm standby** — its internal SPSC queues are populated but its gateway thread doesn't submit orders. Failover is a matter of microseconds, not seconds.

### 9. Summary: The Horizontal Scaling Mental Model

| Concern | Solution |
|---|---|
| **Inter-node transport** | RDMA one-sided writes replicating SPSC semantics |
| **Fan-out to many machines** | Hardware multicast (IB) or DPDK multicast UDP |
| **Workload decomposition** | Shard by exchange, instrument group, or strategy type |
| **MPSC at network boundary** | One local SPSC queue per remote source on the receiver |
| **Clock sync** | PTP with hardware timestamping or GPS-disciplined oscillators |
| **Physical proximity** | Same rack → DAC cable; same city → dark fiber; cross-city → microwave |
| **Fault tolerance** | Shadow clusters consuming RDMA copies of each SPSC output |

The mental model is this: **horizontal scaling stretches the SPSC chain across machines**, with RDMA as the wire between nodes. Each machine remains a self-contained set of lock-free SPSC pipelines internally. The art of HFT horizontal scaling is ensuring the inter-machine hops are so fast and so transparent that the distributed pipeline behaves as close as possible to a single-machine SPSC chain.
