## Vertical Scaling for HFT with SPSC

Vertical scaling in HFT means extracting maximum performance from a **single physical machine**. Unlike horizontal scaling (more machines), vertical scaling focuses on hardware topology awareness, minimizing latency variance, and ensuring your SPSC pipelines are never bottlenecked by the hardware layer beneath them.

---

### 1. CPU Selection & Topology

#### Choose the Right Die
The ideal HFT CPU is **not** the one with the most cores — it's the one with the **lowest and most consistent core-to-core latency**.

| CPU Family | Notes |
|---|---|
| **Intel Xeon (single socket)** | Preferred. Single-socket eliminates NUMA entirely. Golden Cove / Sapphire Rapids cores have excellent store-forwarding latency |
| **AMD EPYC** | CCX-to-CCX latency is a trap — keep SPSC thread pairs within the same CCD |
| **Intel Core (i9)** | Used in ultra-low-latency shops; fewer cores, higher clocks, simpler topology |

**The golden rule:** the producer and consumer of an SPSC queue must **always run on cores that share an L3 cache slice** (or at minimum, are on the same physical die).

#### Core-to-Core Latency is Everything
When the producer writes to the ring buffer and the consumer reads it, the data travels through the cache coherence fabric. You want this to be **40–80 ns**, not 150–300 ns (cross-socket NUMA).

```
Producer Core  →  L1 miss  →  L3 (shared)  →  Consumer Core
                   ~4 cycles     ~30 cycles      ~4 cycles
```

Mapping SPSC thread pairs to **physically adjacent cores** on the same LLC slice is the single highest-leverage vertical scaling decision.

---

### 2. CPU Isolation & Thread Pinning

Raw hardware means nothing if the OS keeps migrating threads. You must remove the OS from the picture entirely on your critical cores.

#### Kernel Boot Parameters
```
isolcpus=2-15          # Remove cores 2-15 from scheduler
nohz_full=2-15         # Stop timer interrupts on those cores
rcu_nocbs=2-15         # Move RCU callbacks off critical cores
irqaffinity=0,1        # Force all IRQs to cores 0-1 only
```

#### Thread Pinning per SPSC Stage
```cpp
// Pin each thread in the SPSC chain to a specific core
cpu_set_t cpuset;
CPU_ZERO(&cpuset);
CPU_SET(core_id, &cpuset);
pthread_setaffinity_np(thread, sizeof(cpu_set_t), &cpuset);

// Set SCHED_FIFO to prevent preemption
struct sched_param param { .sched_priority = 99 };
pthread_setschedparam(thread, SCHED_FIFO, &param);
```

#### NUMA Binding (if multi-socket is unavoidable)
```bash
numactl --cpunodebind=0 --membind=0 ./trading_engine
```
All SPSC queues and the threads that touch them must be **pinned to the same NUMA node**. Cross-node memory access doubles or triples your latency.

### 3. Memory Architecture

#### Huge Pages
The SPSC ring buffer must **never suffer a TLB miss** during operation. Allocate it with 2MB (or 1GB) huge pages:

```cpp
void* allocate_spsc_buffer(size_t size) {
    void* ptr = mmap(nullptr, size,
                     PROT_READ | PROT_WRITE,
                     MAP_PRIVATE | MAP_ANONYMOUS | MAP_HUGETLB,
                     -1, 0);
    mlock(ptr, size);  // Pin to RAM — no page faults ever
    return ptr;
}
```

`mlock()` is non-negotiable. A page fault on a cache miss during trading is a **multi-microsecond disaster**.

#### Cache Line Alignment & Padding
The head and tail of the SPSC queue must live on **separate cache lines** to prevent false sharing:

```cpp
struct alignas(64) SPSCQueue {
    // Producer cache line
    alignas(64) std::atomic<uint64_t> head{0};
    char pad1[56];  // fill to 64 bytes

    // Consumer cache line  
    alignas(64) std::atomic<uint64_t> tail{0};
    char pad2[56];

    // The ring buffer itself — on huge pages
    T* buffer;
    uint64_t mask;
};
```

Without this, the producer writing `head` and the consumer reading `tail` will bounce a shared cache line between cores — effectively serializing them despite there being no logical dependency.

#### DRAM Channel Saturation
Ensure your ring buffers are spread across **all memory channels** on the socket. A system with 8 channels running buffers only on channel 0 wastes 7/8 of available bandwidth. Use `numactl --hardware` to verify interleaving.

### 4. NIC & Kernel Bypass

This is where vertical scaling has the most dramatic impact on the inbound SPSC path (NIC → Market Data Handler).

#### Kernel Bypass Stack
| Technology | Latency | Notes |
|---|---|---|
| **DPDK** | ~1–2 µs | Poll-mode driver; CPU core dedicated to spinning |
| **Solarflare OpenOnload** | ~1–3 µs | Kernel bypass TCP; drop-in for standard sockets |
| **Mellanox/NVIDIA RDMA** | < 1 µs | For co-location direct links |
| **FPGA (Xilinx/Altera)** | < 100 ns | Pre-parse on hardware before CPU ever sees packet |

The NIC's poll thread **is** the SPSC producer for the first queue in the chain. Dedicate a full isolated core to it — it must spin 100% of the time (busy-wait, no sleep).

#### RSS & Queue Affinity
Configure **Receive Side Scaling** to pin each NIC hardware queue to a specific core:
```bash
# Pin NIC queue 0 IRQ to core 2 (your market data core)
echo 4 > /proc/irq/<irq_number>/smp_affinity  # bitmask for core 2
```
This ensures packets from a specific exchange feed land directly in the L1 cache of the core that will produce into the SPSC queue — eliminating any cross-core cache traffic on ingress.

### 5. Scaling the SPSC Pipeline Across More Cores

Vertical scaling means you have more cores available. The pattern is to **decompose your pipeline into finer stages**, each connected by an SPSC queue, and assign each stage its own dedicated isolated core.

```
Core 2          Core 3            Core 4          Core 5          Core 6
  │                │                 │               │               │
[NIC Poll]──SPSC──[MD Parser]──SPSC──[Order Book]──SPSC──[Signal]──SPSC──[OMS/GW]
```

As you add cores (vertical scale-up), you can:
- **Split feeds**: dedicate one Core→Core SPSC pipeline per exchange feed
- **Parallelize instruments**: one order book core per symbol group, fed by a dispatcher (still SPSC per consumer)
- **Add a risk shadow**: fan out OMS events into a second SPSC to a dedicated risk core without slowing the main path

#### Fan-Out Pattern (One Producer → Multiple Consumers)
When you need to send the same event to multiple downstream cores (e.g., book update to both signal engine and risk monitor), use **multiple independent SPSC queues** — one per consumer — rather than an MPMC queue:

```
[Order Book Core]
        │
        ├──SPSC──► [Signal Engine Core]
        │
        └──SPSC──► [Risk Monitor Core]
```

This preserves the lock-free, wait-free properties of SPSC on every path.

### 6. Clock & Timing Infrastructure

#### TSC-Based Timestamping
Use `RDTSC` directly — never `clock_gettime()` on the hot path. With modern CPUs, TSC is synchronized across cores on the same socket (invariant TSC):

```cpp
inline uint64_t rdtsc() {
    uint32_t lo, hi;
    __asm__ __volatile__("rdtsc" : "=a"(lo), "=d"(hi));
    return ((uint64_t)hi << 32) | lo;
}
```

#### Turbo Boost & C-States
Disable them both. Variable clock frequency destroys latency predictability:
```bash
# Disable C-states (keep cores in C0 always)
for i in /sys/devices/system/cpu/cpu*/cpuidle/state*/disable; do echo 1 > $i; done

# Set performance governor (max frequency, fixed)
cpupower frequency-set -g performance

# Disable turbo
echo 1 > /sys/devices/system/cpu/intel_pstate/no_turbo
```

A core waking from C1 sleep adds **microseconds** of jitter — unacceptable when your entire SPSC pipeline budget is measured in hundreds of nanoseconds.

### 7. OS & Kernel Tuning

| Tuning | Command / Config | Reason |
|---|---|---|
| Disable hyper-threading | BIOS setting | Sibling threads share L1/L2, polluting the cache of your HFT thread |
| Disable NUMA balancing | `echo 0 > /proc/sys/kernel/numa_balancing` | Prevents OS from migrating pages |
| Increase `vm.swappiness` to 0 | `sysctl vm.swappiness=0` | No swapping, ever |
| Disable transparent huge pages | `echo never > .../transparent_hugepage/enabled` | THP compaction causes latency spikes |
| Preemption model | `CONFIG_PREEMPT_NONE` kernel build | Server kernel, minimal preemption points |
| IRQ coalescing off | `ethtool -C eth0 rx-usecs 0` | Don't batch NIC interrupts |

### Summary: The Vertical Scaling Mental Model

Think of vertical scaling for SPSC-based HFT as **buying more lanes on a private highway and removing every traffic light**:

- **More cores** = more SPSC pipeline stages, finer decomposition, more parallel feeds
- **Better CPU** = shorter core-to-core cache coherence latency between producer and consumer
- **More RAM channels** = ring buffers never starve for memory bandwidth
- **Faster NIC + kernel bypass** = the first SPSC producer gets data sooner
- **OS isolation** = no unexpected cars entering your private highway

Every layer of the hardware stack must be configured to serve the SPSC chain — not the general-purpose OS workload the hardware was designed for by default.
